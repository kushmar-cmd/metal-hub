📋 תהליך שינוי הנדסי — ECO/ECN
1. בקשת שינוי (ECR)
  · גורם מגיש: הנדסה / ייצור / לקוח
  · תיאור שינוי + נימוק + השפעות
  · מספר חלק + Rev נוכחי
2. ניתוח השפעה
  · חלקים מושפעים (Where-Used)
  · מלאי קיים — מה לעשות עמו?
  · עלות + לוחות זמנים
3. אישור (ECO Board / CCB)
  · הנדסה + ייצור + רכש + אחסון
  · תאריך כניסה לתוקף (Effectivity)
4. יישום
  · עדכון שרטוט + Rev
  · עדכון BOM / דרישות
  · הודעה לכל המחלקות (ECN)
5. סגירה ותיוק
  · רשומה ב-PLM/ERP
  · שמירת Baseline
ECR — Engineering Change Request
הפתח לתהליך: מי מבקש, מה הבעיה (כשל שטח? הוזלה? ספק?), פתרון מוצע, הערכת השפעה. ה-CCB מאשר/דוחה.
ECO — Engineering Change Order
ההוראה המחייבת: מה משתנה + Rev חדש, Effectivity (S/N/תאריך), טיפול במלאי קיים (Use-As-Is/Rework/Scrap).
ECN — Engineering Change Notice
ההפצה הרשמית לכל הנוגעים — ייצור, רכש, איכות, ספקים. מוודאת שכולם עובדים על אותה גרסה.
🔎 Where-Used — ניתוח השפעת שינוי
לפני כל ECO — השאילתה הראשונה: איפה עוד החלק חי? שינוי בבורג "פשוט" שמופיע ב-40 מכלולים = 40 בדיקות השפעה נדרשות. PLM נותן תשובה בלחיצה; ניהול ידני בקבצים = הזמנה לתקלה.

Redline — תיקון בעט אדום
סימון ידני זמני על עותק שרטוט בשטח — עד עדכון רשמי. חוקי רק עם חתימה מוסמכת + תאריך + ECO פתוח בעקבותיו!
As-Built
מצב סופי בפועל — כולל סטיות מאושרות, החלפות חומר, ותיקוני שטח. קריטי למסירת פרויקטים ותחקירי כשל.
Interchangeability — FFF
Form-Fit-Function: שינוי ששומר FFF = אותו מק"ט + Rev חדש. שובר FFF = חובה מק"ט חדש!

💾 PLM — Product Lifecycle Management
המערכת שמחזיקה הכל במקום אחד: קבצי CAD עם גרסאות, BOM, תהליכי ECO אלקטרוניים, חתימות דיגיטליות. היתרון המרכזי: מקור אמת אחד — אין "גרסה על שולחן של מישהו".
PDM — Product Data Management
הכספת של קבצי CAD: Check-in/out (אין דריסות!), גרסאות אוטומטיות, קשרי הרכבה-חלק. SolidWorks PDM נפוץ.
PLM המלא
מרעיון ועד הוצאה משירות: דרישות, CAD, BOM, ECO, ייצור, שירות. Windchill, Teamcenter, 3DEXPERIENCE.
📐 Configuration Management — ניהול תצורה
Baseline — נקודת עיגון תצורה מאושרת. סוגים: FBL (Functional — דרישות), ABL (Allocated), PBL (Product — התיעוד המלא לייצור). מרגע הקיבוע — כל שינוי רק דרך ECO.

Configuration Audit — משווה: מה מתועד (Baseline) מול מה שנבנה בפועל (As-Built). חובה לפני מסירת פרויקט קריטי.
Deviation — סטייה מאושרת מראש
"אנחנו יודעים שלא נעמוד בדרישה X — אפשר?" → ניתוח הנדסי → אישור לפני הייצור, לאצווה/כמות מוגדרת.

דוגמה: קוטר חור בגלם חורג ב-0.05mm — מאושר מראש כי לא משפיע על הרכבה.
Waiver — אישור בדיעבד
"יצא חריג — מה עושים?" → MRB בודק → אישור אחרי שהחלק כבר יוצר חריג.

דוגמה: חלק שכבר יוצר עם גימור משטח מעט גס מהנדרש — MRB בודק אם זה עדיין תקין פונקציונלית.
⚠️ כללי זהב
• שני המסמכים מוגבלים בכמות/זמן — אינם משנים את השרטוט הרשמי!
• תיעוד מלא חובה — לא "החלטה בעל פה"
• שימוש חוזר/קבוע באותה חריגה = דגל אדום — זה הזמן להוציא ECO אמיתי ולעדכן את השרטוט
• Deviation/Waiver שחי חודשים בלי ECO בעקבותיו = פצצה מתקתקת של גרסאות סותרות
🔍 MRB — Material Review Board
הפורום שמחליט על גורל חלקים חריגים — לא עומדים בדרישות השרטוט. הרכב: הנדסה + איכות (+ לקוח, בפרויקטים ביטחוניים/קריטיים).
החלטהמשמעותדורש
Use-As-Is (UAI)טוב כמו שהוא — למרות החריגההצדקה הנדסית מתועדת!
Reworkעיבוד נוסף להחזרה לתוך הדרישהתהליך מאושר, בדיקה חוזרת
Repairתיקון (לא זהה למקורי, אבל תקין)תהליך תיקון מאושר במיוחד
Scrapפסילה מוחלטתתיעוד + השמדה/סימון
⚠️ סמכות UAI (Use-As-Is) — רק למוסמכים! MRB Crib = אזור פיזי לחלקים חורגים הממתינים להחלטה, מונע שהם "יתגלגלו" בטעות לשרשרת הייצור.

CAGE Code
קוד 5 תווים המזהה ספק ביטחוני עולמית. מופיע בכותרת שרטוטים, CoC, פקודות רכש. ישראל: DSPO. ארה"ב: DLIS.
Effectivity + ITAR
שרטוט ITAR = "יצוא" גם במייל לגורם זר! גישה מוגבלת ל-US Persons או ברישיון. הפרה = קנסות ומאסר.
